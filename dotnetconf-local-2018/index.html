Background

API Security Timeline

  • 👹 until 2010: SAML & others
    The dark ages
  • 🥚 2010: OAuth 1.0
    RFC 5849. Open standard for API access delegation
  • 🐣 2012: OAuth 2.0 RFC 6749. Not backwards compatible with OAuth 1.0
  • 🐥 2014: OpenID Connect
    Identity layer on top of OAuth 2.0
  • 📝 2015: JSON Web Token
    RFC 7519. Token & claim assertions standard
  • 🛡 soon: Token Binding
    Cryptographically binding tokens to TLS connection

Standards

OAuth 2.0

  • 🛠 Four flows (different use cases)
    authorization code (most common), implicit (JS), client credentials (M2M), password (legacy)
  • 👜 Bearer concept Authorization: Bearer {token}
  • 🎭 Public client concept
    Implicit grant, optional secret, no backend brokerage
  • 🤖 Delegated authorization only
    Does not deal with authentication, only describes granting authorization

OAuth 2.0 Flows

Flow Interactive Use case
client_credentials ❌ machine-to-machine communication
no subject, no refresh_token
password ❌ password brokerage for legacy apps
only for 100% trusted clients - user hands over the password to the client
implicit ✅ public JavaScript clients
no backend involvement, no refresh_token
authorization_code ✅ most common
similar to implicit, but with backend involvement

OpenID Connect

  • 🏭 OAuth 2.0 extension
    Adds identity concept and federated authentication
  • 🦄 Identity token id_token in addition to access token
  • 📚 Discovery endpoint
    Standardzied way of exposing server metadata
  • 🖥 Session management
    Log out once

Disocvery endpoint is available at /.well-known/openid-configuration.

It allows clients to retrieve all security metadata, such as token & introspection URLs or JSON Web Keys.

JSON Web Tokens (JWT) is an open standard (RFC 7519) method for representing claims securely between two parties.

id_token payload structure

    
{
    "nbf": 1539532647,     // not before timestamp
    "exp": 1539532947,     // expires timestamp
    "iss": "https://localhost:44356/openid",     // issuer
    "aud": "dummy_user_app_client",     // audience (maps to CLIENT ID only)
    "iat": 1539532647,     // issued at timestamp
    "at_hash": "n3nAEb6kcWnM53ls0vJ-bQ",     // access token hash
    "sub": "d01440edc505489e904878ce9738865a",     // subject
    "auth_time": 1539532259,     // authentication time (can be different than iat)
    "idp": "demo_idp",     // identity provider name
    "amr": [     // authentication method reference
      "pwd"
    ]
}

access_token payload structure

    
{
    "nbf": 1539532647,     // not before timestamp
    "exp": 1539536247,     // expires timestamp
    "iss": "https://localhost:44356/openid",     // issuer
    "aud": [     // resource APIs that can consume this token
      "dev.contacts.api"
    ],
    "client_id": "dummy_user_app_client",     // client for which the token was issued
    "sub": "d01440edc505489e904878ce9738865a",     // subject
    "auth_time": 1539532259,     // authentication time
    "idp": "embedded_idp",     // identity ptovider name
    "email": "filip@w.com",     // custom claim
    "scope": [     // scopes
      "openid",
      "contacts.view",
      "offline_access"
    ],
    "amr": [     // authentication method reference

    ]
}

Authentication

API Authentication anti-patterns

  • 💩 Rolling out custom security
    Re-inventing the wheel
  • 💩 Using implicit authentication
    i.e. classic cookie-authentication models
  • 💩 Mixing authN and authZ
    i.e. only OAuth 2.0 for both purposes
  • 💩 Fat tokens
    Transporting too much data in tokens

Authentication in ASP.NET Core

  • ✅ Authentication handlers
    Registered in DI - cookies, social, JWT, OpenIdConnect
  • ✅ Middleware invoked
    Can also be manually triggered
  • ✅ Claims transformation
    Allows per-request manipulation of claims
  • ✅ Based loosely on Katana
    Completely new if you have not worked with OWIN

Adding token validation to ASP.NET Core

    
public void ConfigureServices(IServiceCollection services)
{
    services.AddMvc();
    services.AddAuthentication("Bearer").AddIdentityServerAuthentication(o =>
        {
            o.Authority = "https://my.identityserver.io/openid";
            o.SupportedTokens = SupportedTokens.Jwt;
        });
}

public void Configure(IApplicationBuilder app, IHostingEnvironment env)
{
    app.UseAuthentication();
    app.UseMvc();
}

Authorization

API Authorization Levels

Level Mechanism Source
⛅ Accessing an API endpoint 🔒 authorization framework access_token scopes
Can you interact with an API in any way?
🌤 Role / permission verification 🔒 authorization framework profile / policy service
Can you invoke a specific operation in the API? For example: send a message.
☀️ Business workflow rules 🗂 business logic custom
Can you invoke a specific operation, with specialized conditions in the API? For example: send a message to user X.

Until ASP.NET Core, ASP.NET never shipped with a comprehensive authorization framework.

Authorization in ASP.NET Core

  • ✅ new IAuthorizationService
    Decouple authorization from business code
  • ✅ Policy based
    No more coupling or misinterpreting "roles"
  • ✅ Requirements concept
    Decouple authorization input from authorization logic
  • ✅ Decoupled from MVC
    works with raw ASP.NET Core, DI integrated

Policy based authorization in ASP.NET Core

        
public void ConfigureServices(IServiceCollection services)
{
    services.AddMvc();
    services.AddAuthorization(o =>
    {
        o.AddPolicy("ViewOrders", 
            p => p.RequireAuthenticatedUser().RequireClaim("scope", "orders.view"));
    });
}

[HttpGet("")]
[Authorize("ViewOrders")]
public async Task GetAll()
{
    // omited for brevity
}

Requirement authorization in ASP.NET Core

        
public class VerifiedCustomerRequirement : IAuthorizationRequirement 
{
    public int OrderTotalThreshold { get; set; } = 10000;
}

public class VerifiedCustomerHandler : AuthorizationHandler
{
    protected override Task HandleRequirementAsync(AuthorizationHandlerContext context,
        VerifiedCustomerRequirement requirement, Order orderResource)
    {    
        if (order.Total < requirement.OrderTotalThreshold || context.User.GetClaim("IsVerifiedCustomer"))
        {
            context.Succeed(requirement);
        }
                    
        return Task.CompletedTask;
    }
}
