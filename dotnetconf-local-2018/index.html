Background

API Security Timeline

  • 👹 until 2010: SAML & others
    The dark ages
  • 🐣 2010: OAuth 1.0
    RFC 5849. Open standard for API access delegation
  • 🐥 2012: OAuth 2.0 RFC 6749. Not backwards compatible with OAuth 1.0
  • 👮 2014: OpenID Connect
    Identity layer on top of OAuth 2.0
  • 📝 2015: JSON Web Token
    RFC 7519. Token & claim assertions standard
  • 🛡 soon: Token Binding
    Cryptographically binding tokens to TLS connection

Standards

OAuth 2.0

  • 🛠 Four flows (different use cases)
    authorization code (most common), implicit (JS), client credentials (M2M), password (legacy)
  • 👜 Bearer concept Authorization: Bearer {token}
  • 🎭 Public client concept
    Implicit grant, optional secret, no backend brokerage
  • 🤖 Delegated authorization only
    Does not deal with authentication, only describes granting authorization

OpenID Connect

  • 🏭 OAuth 2.0 extension
    Adds identity concept and federated authentication
  • 🦄 Identity token id_token in addition to access token
  • 📚 Discovery endpoint
    Standardzied way of exposing server metadata
  • 🖥 Session management
    Log out once

JSON Web Tokens (JWT) is an open standard (RFC 7519) method for representing claims securely between two parties.

Authentication

API Authentication anti-patterns

  • 💩 Rolling out custom security
    Re-inventing the wheel
  • 💩 Using implicit authentication
    i.e. classic cookie-authentication models
  • 💩 Mixing authN and authZ
    i.e. only OAuth 2.0 for both purposes
  • 💩 Fat tokens
    Transporting too much data in tokens

Authentication in ASP.NET Core

  • ✅ Authentication handlers
    Registered in DI - cookies, social, JWT, OpenIdConnect
  • ✅ Middleware invoked
    Can also be manually triggered
  • ✅ Claims transformation
    Allows per-request manipulation of claims
  • ✅ Based loosely on Katana
    Completely new if you have not worked with OWIN

.NET introduced ClaimsPrincipal as primary identity model in 2012 (.NET 4.5).

Authorization

API Authorization Levels

Level Mechanism Source
⛅ Accessing an API endpoint 🔒 authorization framework access_token scopes
Can you interact with Orders API in any way?
🌤 Role verification 🔒 authorization framework profile / policy service
Can you invoke a specific operation in the Orders API? For example: place a new order.
☀️ Business workflow permissions 🗂 business logic custom
Can you invoke a specific operation, with specialized conditions in the Orders API? For example: place a new order with value over 10 000 EUR.

Until ASP.NET Core, ASP.NET never shipped with a comprehensive authorization framework.

Authorization in ASP.NET Core

  • ✅ new IAuthorizationService
    Decouple authorization from business code
  • ✅ Policy based
    No more coupling or misinterpreting "roles"
  • ✅ Requirements concept
    Decouple authorization input from authorization logic
  • ✅ Decoupled from MVC
    works with raw ASP.NET Core, DI integrated
