Warm up

Symmetric encryption

The most popular symmetric algorithm is the Advanced Encryption Standard (AES)

AES with 256 bit key length is approved by NSA for top secret information.

RSA is considered to be partly legacy, but still it makes a good example to illustrate certain concepts - because they are applicable to the alternative algorithms as well.
Symmetric encryption is fast and relatively simple, but requires a secure method for key distribution, which is a big challenge.

Asymmetric encryption

Also known as Public Key Cryptography. It is also used for digital signatures.
  • 🔒 Key exchange/establishment
    → establishing a secure communication channel
    → then hand over to symmetric encryption for efficiency
  • ✍️ Digital signatures
    → create and verify signatures
    → serve as proof of digital content authenticity and integrity
Many protocols - for example TLS/SSL - use a hybrid model - start with an asymmetric handshake, and then switch to symmetric.

Asymmetric cryptography is based on difficult mathematical problems:

  • 🧮 factorizing large composite numbers - RSA
  • 🧮 elliptic curve discrete logarithm problem - EC Cryptography

Quantum threats

In quantum theory a basic unit of information is called a

qubit

Just like a bit, qubit has only two observable (basis) states

$ \ket{0} \rightarrow $ maps to bit 0

$ \ket{1} \rightarrow $ maps to bit 1

Mathematically, a qubit state is described by a vector.
Schrödinger's cat was both dead and alive at the same time. It's NOT that he was dead or alive and Schrödinger simply did not know. Only once Schrödinger opened the box, because of "observation", would the cat's fate be decided.

Program evolution

type input execution output
Classical 010 $\rightarrow$ 110 $\rightarrow$ 001
Quantum $\ket{1}\ket{0}\ket{0}$ $\rightarrow$ $\ket{\psi}$ $\rightarrow$ $\ket{0}\ket{1}\ket{0}$

The strength of quantum computers is that they are able to take algorithmic advantage of superposition. This is called

quantum parallelism

  1. Create a superposition of all possible results
  2. Use interference to amplify correct result, and cancel out bad results

source: Stefano Markidis, KTH Royal Institute of Technology, https://arxiv.org/abs/2405.07222

...quantum parallelism allows quantum computers to evaluate a function $f(x)$ for many different values of x simultaneously.

- Nielsen and Chuang

One of most widely used public-private cryptosystems is RSA (Rivest–Shamir–Adleman)

Its security relies on the factoring problem of large prime numbers.

RSA is considered to be partly legacy, but still it makes a good example to illustrate certain concepts - because they are applicable to the alternative algorithms as well.

RSA

  • 1. Pick two huge prime numbers $p$ and $q$
  • 2. Pick exponent $e$
  • 3. Publish $N = p \times q$ and $e$
  • 4. Encrypt:
    $C = M^e (mod\;N)$     (public key based on $N$)
  • 5. Decrypt:
    $M = C^{e^{-1} (mod\;(p-1) \times (q-1))} (mod\;\;N)$    (private key based on $p$, $q$)
The value of e is almost always fixed to 65537.

Effectively, the security of RSA boils to the fact that

finding $p$ and $q$ is impossible

if you only know $N$.

In 1994, Peter Shor developed an integer factorization algorithm for quantum computers that achieves exponential speed up over the fastest known classical alternative.

Factoring a number $N$

  • 1. Pick $a$ such that $a < N$ and that it is relatively prime to $N$
  • 2. Find the period $r$ of a function $f(x) = a^x (mod\;\;N)$
  • 3. Calculate:
    $p = gcd(a^{\frac{r}{2}} + 1, N)$ and $q = gcd(a^{\frac{r}{2}} - 1, N)$

Example: factoring 15

  • 1. Set $a = 13$
  • 2. Find order $r = 4$
  • 3. Calculate:
    $p = gcd(170, 15) = 5$
    $q = gcd(168, 15) = 3$

5 and 3 are factors of 15.

Shor's algorithm relies on inverse Quantum Fourier Transform used for function order finding. QFT is exponentially faster than any classical alternative.

Shor's algorithm is a hybrid classical-quantum method - gcd is still done classically.
(...) attacking elliptic curve cryptography is indeed easier than attacking RSA (...)
Forward secrecy means using new (ephemeral) keys all the time, so intercepting and breaking one message would not reveal the whole session. ECDH is also used in tools like Signal or Skype.

The best current estimation for Shor's algorithm resource requirements for breaking RSA-2048 are¹:

  • 🧮 1,399 logical qubits
     → ideal qubits - not physically attainable, suitable for mathemtical calculation only
  • 🧮 897,864 physical qubits
     → assuming a surface code architecture with a 0.1% physical gate error rate.


¹ C. Gidney, "How to factor 2048 bit RSA integers with less than a million noisy qubits," arXiv:2505.15917 (2025).

OK, that looks worrying, but thankfully we still have some time, right?

No, it is already too late

In TLS 1.3, RSA is no longer used for key exchange, in favor of methods like Ephemeral Elliptic-Curve Diffie-Hellman (ECDHE). Even without TLS 1.3, ECDSA should be used over RSA.
With a harvest-now-decrypt-later approach, a threat actor can store encrypted data now and decrypt it later when quantum computers are available.
Every day that you don't convert to a quantum-safe protocol, there's no recovery plan

- Dario Gil, IBM senior VP

In 1996, Lov Grover developed a quantum search algorithm through an unsorted data set - finding $x$ for $f(x) = y$, when $y$ is known.

It provides quadratic speedup over the classical counterparts.

Because of that it can be used to attack symmetric encryption schemes such as AES, however that can be easily countered by increasing the key size.

Brute force AES 256:

Post quantum cryptography

In 2016 NIST (National Institute of Standards and Technology) announced a post-quantum cryptography competition, with the aim to provide a standardized crypto scheme resistant to quantum threats.

In 2022 the four winners have been announced.

  • 🔐 CRYSTALS-Kyber
    asymmetric encryption, lattice-based
  • ✍️ CRYSTALS-Dilithium
    signature, lattice-based
  • ✍️ FALCON
    signature, lattice-based
  • ✍️ SPHINCS+
    signature, hash-based
Kyber & Dilithium were selected to be primary general-purpose protocols for key encapsulation and digital signatures. FALCON was recommended for applications requiring smaller digital signatures. SPHINCS+ was chosen as a backup option to the lattice-based approaches.

$t = (\braket{\mathbf{a}, \mathbf{s}} + \varepsilon) \mod q$

  • · $\braket{\mathbf{a}, \mathbf{s}}$ is the dot (inner) product of two vectors
  • · $\varepsilon$ is the error term, chosen from a specific distribution
  • · $q$ is the modulus, confining the value to a predictable range
  • · $t$, $q$, and $\mathbf{a}$ are published, but $\mathbf{s}$ and $\varepsilon$ are kept secret

note: this is a simplified picture of LWE, for illustrative purposes only

The security in LWE comes from the fact that, while it is straightforward to compute $t$ given $\mathbf{a}$, $\mathbf{s}$, $q$, and $\varepsilon$, it is computationally hard to reverse this process. The difficulty arises from the addition of the error term $\varepsilon$, which obscures the relationship between $t$ and $\mathbf{s}$, making it infeasible to recover $\mathbf{s}$ given $t$, $\mathbf{a}$, and $q$.

In 2024 the first three PQC standards were formalized.

  • 🔐 ML-KEM (Kyber)
    → Module-Lattice-Based Key-Encapsulation Mechanism Standard
    → FIPS-203
  • ✍️ ML-DSA (Dilithium)
    → Module-Lattice-Based Digital Signature Standard
    → FIPS-204
  • ✍️ SLH-DSA (SPHINCS+)
    → Stateless Hash-Based Digital Signature Standard
    → FIPS-205

Credit: N. Hanacek/NIST

Kyber & Dilithium were selected to be primary general-purpose protocols for key encapsulation and digital signatures. FALCON was recommended for applications requiring smaller digital signatures. SPHINCS+ was chosen as a backup option to the lattice-based approaches.

In late 2024 NIST published the official deprecation timeline

  • ⚠️ 2030: Deprecation of 112-bit security algorithms
    RSA-2048, ECC P-224 and lower
  • ⛔ 2035: Disallowing all non-quantum-resistant algorithms
    → All ECC and RSA is disallowed!

There are a number of ongoing IETF standardization efforts for PQC:

  • 🤝 draft-ietf-tls-ecdhe-mlkem
    → Hybrid key exchange for TLS 1.3, combining classical and PQC algorithms (X25519MLKEM768).
    → This is supported approach in all major browsers today.
  • 🚀 draft-ietf-tls-mlkem
    → Pure post-quantum key exchange for TLS 1.3.
    → This is the long-term goal, not yet widely supported.
  • 🛠️ draft-ietf-hpke-pq
    → PQC for a general encryption framework (HPKE).
    → Used in other protocols like Encrypted Client Hello and secure messaging.

In September 2023 Signal was the first messaging app to move to PQC - their own Post-Quantum Extended Diffie-Hellman (PQXDH) protocol based on Kyber.

In 2024 Apple updated iMessages with PQ3, the strongest PQC protocol, with support for regular rekeying.

Source: Apple, https://security.apple.com/blog/imessage-pq3/

In April 2025 OpenSSL 3.5 was released, with official support for ML-KEM, ML-DSA and SLH-DSA. It's an LTS released and it paves the way for wide adoption of PQC in all Linux-based systems.

In May 2025 Microsoft released initial early support of PQC for Windows - however currenly only for the Canary Windows Insiders channel.

Cryptography in .NET

  • 🧩 One common API surface
    → System.Security.Cryptography namespace across all platforms
  • ⚙️ Native backends per OS
    → Windows: CNG / CAPI
    → Linux: OpenSSL
    → macOS/iOS: CommonCrypto
    → Android: JCA / OS crypto
  • 🌐 Third-party managed implementations
    → Portable across platforms (e.g., BouncyCastle)
    → Can implement algorithms not in OS → No hardware acceleration/FIPS compliance
The .NET crypto API is unified, but the runtime delegates to the OS crypto stack for performance, compliance, and integration. This means your code looks the same, but the actual implementation varies by platform.

.NET 10 Preview 6 introduced the first official .NET API for PQC into System.Security.Cryptography. It currently supports:

  • → Windows (CNG): Canary Channel Build 27852+
  • → Linux (OpenSSL): 3.5+

While the official framework support for PQC in .NET is still limited, BouncyCastle has long been providing managed implementations of all NIST PQC algorithms.

Open Quantum Safe has been providing verified quantum-resistant crypto algorithms implementations in C.

LibOQS.NET is a thin .NET wrapper around the Open Quantum Safe library, providing easy access to post-quantum cryptography features for .NET developers.
