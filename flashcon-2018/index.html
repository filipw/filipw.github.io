Can you trust the compiler?

Customers will depend on the new compiler behaving exactly the same way as the old one. Writing a new compiler for C# means trying to match the old compiler bug-for-bug.

- Mads Torgersen, C# Lead Designer

Can the compiler produce a program that won't work?
Yes - for example due to CLR/CoreCLR bugs.

Can the runtime run the code that compiler doesn’t allow?
Yes - CLR is much less restrictive.

Can the compiler produce code that doesn’t behave as expected?
Yes - compiler interprets the language spec.

Can you bypass some of the strict compiler rules?
Yes - compiler has a lot of "knobs".

Authentication

API Authentication anti-patterns

  • 💩 Rolling out custom security
    Re-inventing the wheel
  • 💩 Using implicit authentication
    i.e. classic cookie-authentication models
  • 💩 Mixing authN and authZ
    i.e. only OAuth 2.0 for both purposes
  • 💩 Fat tokens
    Transporting too much data in tokens

Authentication in ASP.NET Core

  • ✅ Authentication handlers
    Registered in DI - cookies, social, JWT, OpenIdConnect
  • ✅ Middleware invoked
    Can also be manually triggered
  • ✅ Claims transformation
    Allows per-request manipulation of claims
  • ✅ Based loosely on Katana
    Completely new if you have not worked with OWIN

Simplest way to add token validation to an ASP.NET Core app is to use IdentityServer4.AccessTokenValidation Nuget package.

Token validation in ASP.NET Core

    
public void ConfigureServices(IServiceCollection services)
{
    services.AddMvc();
    services.AddAuthentication("Bearer").AddIdentityServerAuthentication(o =>
        {
            o.Authority = "https://my.identityserver.io/openid";
            o.SupportedTokens = SupportedTokens.Jwt;
        });
}

public void Configure(IApplicationBuilder app, IHostingEnvironment env)
{
    app.UseAuthentication();
    app.UseMvc();
}

Authorization

Until ASP.NET Core, ASP.NET never shipped with a comprehensive authorization framework.

API Authorization Levels

Level Mechanism Source
⛅ Accessing an API endpoint 🔒 authorization framework access_token scopes
Can you interact with an API in any way?
🌤 Role / permission verification 🔒 authorization framework profile / policy service
Can you invoke a specific operation in the API? For example: send a message.
☀️ Business workflow rules 🗂 business logic custom
Can you invoke a specific operation, with specialized conditions in the API? For example: send a message to user X.

Authorization in ASP.NET Core

  • ✅ new IAuthorizationService
    Decouple authorization from business code
  • ✅ Policy based
    No more coupling or misinterpreting "roles"
  • ✅ Requirements concept
    Decouple authorization input from authorization logic
  • ✅ Decoupled from MVC
    works with raw ASP.NET Core, DI integrated

Policy based authorization in ASP.NET Core

        
public void ConfigureServices(IServiceCollection services)
{
    services.AddMvc();
    services.AddAuthorization(o =>
    {
        o.AddPolicy("ViewOrders", 
            p => p.RequireAuthenticatedUser().RequireClaim("scope", "orders.view"));
    });
}

[HttpGet("")]
[Authorize("ViewOrders")]
public async Task GetAll()
{
    // omited for brevity
}

Requirement authorization in ASP.NET Core

        
public class VerifiedRequirement : IAuthorizationRequirement 
{
    public int OrderLimit { get; set; } = 10000;
}

public class VerifiedCustomerHandler : AuthorizationHandler<VerifiedRequirement, Order>
{
    protected override Task HandleRequirementAsync(AuthorizationHandlerContext context,
        VerifiedRequirement requirement, Order order)
    {    
        if (order.Total < requirement.OrderLimit || context.User.Claim<bool>("IsVerifiedCustomer"))
        {
            context.Succeed(requirement);
        }
                    
        return Task.CompletedTask;
    }
}
